# IrisLime Infrastructure Control & Test Matrix Specification

This document serves as the authoritative architectural specification and ledger tracking guide for the orchestration layers governing local inference on Intel graphics processing architectures.

## 1. Architectural Design Shifts

### 1.1 Object-Oriented Path Management
The repository automation engine has transitioned from unsafe string interpolation rules (`.replace("\\", "/")`) to object-oriented filesystem tracking architectures driven by the Python `pathlib` module. 
* All internal parameters and boundary variables enforce strict POSIX serialization mapping using the explicit `.as_posix()` method.
* This isolates the underlying tooling from path-mangling bugs introduced by cross-subsystem shell switches (e.g., handling spaces and drive-letter colons across Git Bash, Windows NT Command line structures, and raw POSIX subprocess pipes).

### 1.2 Isolated Binary Suffix Scans
To handle host OS variations cleanly, the test suite coordinator dynamically queries `os.name`. When navigating native Windows hosts (`nt`), it appends a structural `.exe` suffix to binary lookup targets like `llama-bench` and `test-backend-ops`. This ensures accurate file-presence tracking loops without cluttering cross-platform runtime checks.

## 2. Compilation and Build Ordering Profiles

To prevent optimization pollution, the compilation engine enforces structured execution constraints. Rather than populating single, volatile shared areas, targets are grouped into separated compilation directories mapped directly to their performance configurations.

### 2.1 Prioritized Compilation Paths
Execution passes must satisfy a strict profile dependency array:
1. **`Debug`** -> Isolated Target Path: `build/<backend>_debug`
2. **`RelWithDebInfo`** -> Isolated Target Path: `build/<backend>_relwithdebinfo`
3. **`Release`** -> Isolated Target Path: `build/<backend>_release`

Compiling debug-enabled profiles first establishes a baseline telemetry signature, helping engineers identify performance regression patterns before high-optimization options run.

### 2.2 Granular Cache Cleansing
To prevent complete workspace rebuild overhead during iterative debugging passes, the system introduces targeted pruning rules. Using commands like `make clean-cache-openvino` surgically purges CMake cache variables (`CMakeCache.txt` and `CMakeFiles/`) while leaving intermediate compiled objects (`.obj`, `.ninja`) intact for fast incremental linkages.

## 3. Testing and Telemetry Infrastructure

### 3.1 Forensic Execution Logging
Every functional test unit or benchmarking loop records execution context maps according to a strict **"log everything, never delete"** paradigm. Volatile terminal data is captured and piped permanently to dedicated log trees (`logs/tests/`).
* **`.log` Master Traces:** Store every raw stdout block, stderr error string, and raw system return value.
* **`.sh` Bash Replicas:** Automatically generate standalone shell scripts tracking every active local environment variable (such as `OpenVINO_DIR` or `PATH`) along with the fully expanded command-line statement to enable instant post-mortem replication.

### 3.2 Time Latency Integration
Every subprocess invocation tracks its latency footprint. The system captures precise wall-clock duration via `time.perf_counter()` down to the millisecond, writing the timing metrics to both forensic log manifests and a cumulative database file: `logs/tests/telemetry_matrix.csv`.

### 3.3 Multi-Parameter Testing Grids
The testing engine evaluates multi-parameter grids defined inside `matrix_control.json`. This enables automated sweeps across variable configurations to locate local hardware saturation points:
* **Context Size Shifts (`-c`):** Iterating through `128`, `512`, and `2048` tokens.
* **Batch Size Scaling (`-b`):** Iterating through `16`, `64`, and `128` steps.
* **GPU Layer Offloading (`-ngl`):** Toggling between complete offloading (`-1`) and raw CPU fallbacks (`0`).
