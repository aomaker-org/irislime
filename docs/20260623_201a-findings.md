## 2. Granular Technical Findings & Telemetry Traces

### Finding Alpha: Container Heterogeneity Verification
Initial validation targeted whether hidden native Brain Floating Point 16 (`BF16`) layer arrays were leaking into the compute stream and causing memory/execution faulting. 

* **The Reality:** File names containing explicit k-quantization tokens (`Q4_K_M`, `Q2_K`, etc.) store over 95% of their weight tensors inside tightly compressed 2-bit to 4-bit integer blocks.
* **Structural Sub-Tensors:** Using structured binary container checks, it was verified that certain sensitive non-quantized topology matrices—such as token embeddings or normalization layernorms—remain at full precision. If the parent base weights originate from a native `BF16` workspace, these minor metadata tensors are preserved as such, potentially forcing hardware emulation paths if the host lacks native 16-bit vector logic support.

### Finding Beta: Interactive IO Pipeline Deadlocks
Early iterations of the automated testing runners (`smoke001.py` to `smoke005.py`) routinely timed out at the strict 10-second ceiling across all compiled CPU configurations.

* **The Cause:** Recent updates to the `llama-cli` utility enforce interactive conversation behavior by default. Even when standard input is closed or passed as `DEVNULL`, the utility outputs generation tokens and immediately drops into a blocking terminal read loop, rendering the prompt symbol (`>`) while waiting for an end-of-file signal or a user carriage return.
* **The Telemetry Solution:** The test design was upgraded to an asynchronous model using `select.select()` to scan output file descriptors concurrently. By intercepting the raw prompt string pattern (`>`) or validation metrics line-by-line, the Python engine can issue an explicit `.terminate()` call to the background runtime, transforming an interactive user application into a stable, scriptable metric-collection tool.

* **The Diagnostic:** The crash occurs entirely out of developer scope inside the system-provided core Level Zero loader loader utility (`libze_loader.so.1`). During standard platform hardware queries, the loader discovers the environment tracing module (`libze_tracing_layer.so.1`) and calls `zelLoaderTracingLayerInit`. Under a virtualized WSL2 boundary layer, the underlying tracking hooks map onto an incomplete virtualized driver interface (`/dev/dxg`), dereferencing a null tracking pointer and forcing an immediate segment violation.
* **The System-Level Fix:** Setting `export ZE_ENABLE_TRACING_LAYER=0` proved insufficient as the loader bypassed the environment state variable during basic initialization sweeps. The definitive resolution requires forcing the Intel OneAPI Unified Runtime to sidestep the Level Zero adapter altogether and route commands through the stable cross-platform OpenCL compute stack:
```bash
export ONEAPI_DEVICE_SELECTOR=opencl:gpu

```


Once implemented, initialization passed, enabling the Iris Xe execution units to process the prefill matrix at a native baseline speed of **12.8 tokens per second**.

### Finding Delta: Microsoft mlas Link-Phase Failure

During parallel assembly compilation of the OpenVINO repository framework, the final shared module linking phase threw a fatal error at the 100% boundary:

```text
/usr/bin/ld: /home/fekerr/src/openvino/bin/intel64/Release/libmlas.a(platform.cpp.o): in function `MLAS_PLATFORM::MLAS_PLATFORM()':

platform.cpp:(.text._ZN13MLAS_PLATFORMC2Ev+0x494): undefined reference to `MlasGemmU8S8KernelAvx512Core'
```

* **The Diagnostic:** The Microsoft Linear Algebra Subprograms component (`libmlas.a`), which operates as a critical backend math layer for the OpenVINO CPU plugin, attempted to reference highly vectorized AVX-512 matrix multiplication kernels (`MlasGemmU8S8KernelAvx512Vnni`). Because the local cross-compilation compiler sequence either stripped these files during code generation or found that the underlying processor's AVX-512 instruction flags were not fully exposed inside this specific WSL2 thread context, the symbol addresses went unresolved.
* **The Resolution Modality:** Since the structural focus of the project centers exclusively on maximizing Iris Xe *integrated graphics processing* rather than secondary CPU-bound thread scheduling, the broken CPU compilation module was entirely pruned from the project tree by passing `-DENABLE_INTEL_CPU=OFF` during the configuration step. This isolated the linkage focus onto the fully completed OpenVINO GPU binaries (`libopenvino_intel_gpu_plugin.so`), forcing a swift, clean compilation exit.
