# IrisLime Quick Start Deployment Recipe

This guide is the friction-free installation sequence for bringing up the IrisLime development environment on a newly cloned machine. Follow these exact terminal steps to link your trees, arm the environment, and execute your first hardware validation test.

## Step 1: Establish Sibling Tree Structure
The workspace requires your custom `llama.cpp` engine fork to sit as a direct sibling to the `irislime` project directory. 

Execute this block to verify your pathing and establish the required development symlinks:

```bash
# Navigate to your central development directory
cd ~/src

# Clone the custom engine fork if it is not already present on this machine
git clone git@github.com:aomaker-org/llama.cpp.git

# Enter your active project workspace
cd ~/src/irislime

# Link the sibling engine fork into the workspace
ln -sf ../llama.cpp llama.cpp

# Create a centralized local storage directory for model binaries and link it
mkdir -p ~/src/ai_models
ln -sf ~/src/ai_models models

```

## Step 2: Initialize the Environment & Dependencies

Initialize the Python virtual environment and pull in required orchestration assets using the master Makefile:

```bash
cd ~/src/irislime

# Initialize the virtual environment and install dependencies
make setup

```

## Step 3: Boot the Shell Session

Always source the private boot utility before compiling or running tests. This loads the Intel oneAPI toolchain compiler variables, activates the Python environment, and transforms your prompt to display active execution history counters:

```bash
. scratch/boot.sh

```

*Note: You must **source** this script using `.` or `source`. Do not execute it directly.*

## Step 4: Fire the Hardware Validation Test

Run the latest unfiltered validation script to verify that your host machine's hardware profile is recognized and that the graphics layer can be reached safely via the OpenCL driver bypass:

```bash
bash scratch/run_test002.sh

```

## Step 5: Verify the Execution Output

Once the test concludes, check that your environment successfully generated your unique hardware logs:

1. **The Raw Log:** Review `scratch/run_test_002_console_*.log` to see the complete, unfiltered GDB terminal stream.
2. **The Results Ledger:** Review `scratch/run_test_002.md` to see your machine's specific CPU identifier and execution status cleanly appended to the tracking table.
3. **The Clipboard:** Verify that the complete terminal execution summary has been copied directly to your clipboard for easy archiving.

# Deep Dive: Getting Started with IrisLime

This document details the underlying system dependencies, host-to-guest hardware bridging mechanics, and structural design patterns governing the IrisLime research environment.

## 1. System Prerequisites

Before initializing the workspace orchestration tools, the host operating system must meet the following hardware and software baselines:

* **OS:** Windows 11 with an active, updated WSL2 (Windows Subsystem for Linux) instance.
* **Toolchain:** Intel oneAPI Base Toolkit installed natively within the WSL2 container (v2026 recommended).
* **Hardware Drivers:** Current Intel Graphics Compute Runtime installation on the Windows host to expose execution layers to virtualized environments.

## 2. The Host GPU Virtualization Bridge

To allow your virtualized Linux container to directly communicate with your integrated Intel Iris Xe silicon, the kernel graphics interface must be initialized.

1. **Expose the Virtual Graphics Module:**
   ```bash
   sudo modprobe vgem

```

2. **Verify Hardware Device Node Availability:**
Confirm that the primary Linux render node is visible and accessible to your user space:
```bash
ls -l /dev/dri/renderD128

```


If this file node is missing, restart your WSL2 instance (`wsl --shutdown` from a Windows prompt) to force the host driver layer to remap the hardware slice.

## 3. Environmental Gating Model

To guarantee absolute repeatability and protect against configuration drift, this workspace completely avoids modifying permanent user configuration files (such as `~/.bashrc`). Instead, everything is loaded dynamically inside your active shell session.

Sourcing the boot pad initializes two critical layers:

```bash
. scratch/boot.sh

```

* **Intel oneAPI compilation variables:** Automatically locates and forces the evaluation of `/opt/intel/oneapi/setvars.sh` to arm the `icx` and `icpx` compilers.
* **Python Isolation:** Activates the localized `.venv` configuration space.

The system verifies readiness using an explicit sentinel marker (`IRISLIME_READY=1`). If this variable is missing from your active environment, the underlying compilation Makefiles will halt execution immediately to prevent unoptimized compilation passes.

## 4. The Isolated Incremental Testing Paradigm

When debugging hardware-level memory faults or driver mismatches across different processor platforms (such as migrating from Core12 to Core11 systems), running unbuffered binaries directly can drop dead with unhandled signals, leaving zero diagnostic evidence behind.

This repository enforces an isolated testing paradigm using sequential, non-destructive scripts located in the `scratch/` directory:

* **Crash Containment:** Binaries are processed via an automated batch-mode GDB execution engine (`scratch/run_gdb.sh`). If a segmentation fault occurs, the parent shell loop catches the true POSIX signal code, logs the stack trace, and terminates gracefully without hanging your terminal window.
* **Privacy Protections:** The testing suite implements an automatic clipboard backup system. Incoming clipboard states are preserved as read-only files inside the untracked `.security_scrub/` workspace before logging tools alter your active buffers.
* **Unfiltered Ledger Mirrors:** Every run records its complete raw console stream to a timestamped file while simultaneously appending clean metrics summaries to a platform result matrix.
